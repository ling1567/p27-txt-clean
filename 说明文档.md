# 27txt 文本格式化工具项目立项说明书

## 1. 项目概述
本项目旨在开发一个基于 Web 界面的文本处理工具，专门用于清理、修复及标准化电子书（特别是 .txt 格式）的内容，提升阅读体验及兼容性。

**目标用户**：
*   **个人读者**：追求高品质离线阅读体验的电子书爱好者。
*   **网文爱好者**：需要清理网页抓取或不规范排版文本的用户。
*   **内容创作者/编辑**：对文本格式有标准化要求的文字工作者。

## 2. 核心功能需求

### 2.1 编码与基础清理

*  **1.读取文件**：
	* 触发文件 I/O，将原始txt文件加载入内存
	*   **文件上限**：最大支持 50MB 的单一 .txt 文件。
	*   **流式处理 (SSE)**：对于超过 10MB 的文件，程序启用流式读取。后端 (FastAPI) 通过 `yield` 将任务进度（百分比及当前状态）实时推送到 SSE 通道，前端同步展示平滑进度条。
	
* **2.编码检测**：
    * 利用探针算法自动识别原始文本的编码格式（全面兼容 GBK, GB18030, Big5, UTF-8, UTF-16 等）。
    * 若检测算法置信度过低或解析失败，系统默认依次按 `UTF-8` 和 `GB18030` 进行静默重试。
    
* **3.UTF-8 转换**：
    * 所有读取到的文件在内存中统一转换为标准的 UTF-8 编码。
    * **容错处理**：在解码过程中遇到无法解析的非法字节时，系统采用“忽略（Ignore）”策略静默丢弃，确保整个大文件解析任务不中断。
    
* **4.BOM 去除：
    * 自动探测并截断文件开头的字节顺序标记（如 UTF-8 的 `EF BB BF` 或 UTF-16 的 `FF FE`），确保输出纯净的无 BOM 文本，防止阅读器解析首行异常。

### 2.2 文本排版与格式标准化
* **5.换行符归一化（默认勾选）**：
    * **最高优先级**：在所有处理步骤的第一步，强制将 Windows 风格的 `\r\n` 和旧版 Mac 风格的 `\r` 统一替换为标准 Unix 换行符 `\n`。这能确保后续所有使用 `^` 和 `$` 的正则表达式行为一致。
* **6.HTML 清理：
    * **标签删除**：通过正则或解析树，一键清除文本中嵌套的 HTML/XML 结构标签（如 `<div>`, `<br/>`, `<span>` 等）。
    * **实体反转义**：将 HTML 实体安全还原为普通字符（例如将 `&nbsp;` 还原为空格，将 `&amp;` 还原为 `&`），保证正文内容的完整性。
    
*   **7.非法字符过滤**：
    *  **安全控制符剥离**：使用正则全局剔除除换行（`\n`, `\r`）和制表（`\t`）以外的所有 C0/C1 控制字符（涵盖 `U+0000` 至 `U+001F`，如 `U+0008` 退格, `U+001A` EOF 等）。  
    * **直接剔除控制字符**： `U+0008`, `U+000B`, `U+000C`, `U+001A` 等。
    *   **清除不可见零宽字符**：`U+200B`, `U+200C`, `U+200D`，`U+FEFF`。
    *   **空白符转换**：将 `U+00A0` 转换为标准半角空格`U+0020`。
    *   **乱码残留处理**：自动剔除解码失败产生的 `U+FFFD`。
    
*   **8.手动过滤非法字符**：（基于 Unicode 识别的交互式清理界面）
	* **UI 唤起**：在左侧配置面板提供一个【手动过滤乱码】的按钮，点击唤出模态弹窗（Modal）。
    *   **异常捕获输入**：支持用户在 UI 界面输入框（一个新的弹框）粘贴包含异常或干扰字符的文本原文（ 200 字以内）。
    *   **字符编码映射**：点击“确定”后，程序自动解析输入片段中的每个字符及其对应的 Unicode 编码（例如：'这' -> `U+8FD9`）。
    *   **可视化交互按钮**：界面将每个字符及其对应的编码生成独立的交互按钮（例如：`[我]` `[是]` `[&!]` `[小]` `[猫]` `[。]`），便于识别不可见字符或乱码。
    *  **不可见字符兜底展示**：若解析出纯粹的控制符或零宽符，按钮表面强制显示其十六进制特征码（如 `[ \u200b U+200B ]`），并在视觉上用特殊颜色高亮，提示用户这就是“隐藏的罪魁祸首”。
    *   **精准剔除逻辑**：用户点击需要删除的字符按钮（如 `[&!]`），程序将该字符对应的 Unicode 编码定义为推入全局状态树 `RuleConfig.base_cleaning.manual_blacklists` 数组中。后续在处理全量文本或动态切片时，该名单内的字符将被全局替换为空字符串 `''`。
    
* **9.全半角转换 ：
    * **精细化控制选项**：抛弃一刀切的转换，提供分类独立切换的配置项：
        1. **字母与数字 (Alphanumeric)**：默认执行**全角转半角**（例如将 `ＡＩ１２３` 转换为 `AI123`），保持英文和数据的紧凑感。
        2. **标点符号 (Punctuation)**：默认执行**半角转全角**（例如将 `, . ? !` 转换为 `， 。 ？ ！`），符合中文阅读习惯。
    * **安全边界保护**：在进行半角转全角标点时，程序需包含基本的豁免逻辑（例如：不处理连续的英文字母内部的标点，以防误伤 URL 网址或英文缩写）。

* **10.段落清洗**：
    * **首尾去杂 (Trim)**：逐行扫描，强制移除每一行开头和结尾的多余空白字符（包括半角空格 `U+0020`、全角空格 `U+3000` 以及制表符 `\t`）。
    * **标准中文缩进 (可选扩展)**：在首尾去杂之后，提供“强制段首缩进”开关，若开启，则在每个非空行的行首自动添加两个全角空格（`\u3000\u3000`）。

* **11.空行压缩 **：
    * **伪空行识别**：定义“空行”不仅包含真正的零字符行（`\n\n`），还包含仅有空格或制表符组成的“伪空行”。处理前需先将伪空行降维成真正的空行。
    * **阈值压缩逻辑**：允许用户自定义保留的空行阈值 N（默认配置 N=2）。程序将连续出现超过 N 个的空行，统一下压折叠为准确的 N 个空行。（若 N=0，则实现“紧凑排版”，消除段落间的所有空隙）。

* **12.章节识别**
	* **内置通用规则库 (Built-in Heuristics)**：
	    * **多形态数字兼容**：默认支持识别“第X章”、“第X节”、“第X回”、“Chapter X”等高频范式。底层的正则匹配必须同时兼容**阿拉伯数字**（1, 2, 3）、**中文数字**（一、二、十、百、千、零）以及常见的**罗马数字**（I, II, III）。
	    * **强制行长防误判机制 (Line-Length Threshold)**：为避免误杀正文，增加硬性约束条件：只有当命中规则的该行**总字符数 ≤ 35 个字符**（阈值可配）时，才被确认为合法章节标题。彻底规避类似“*第一章的内容他早就背得滚瓜烂熟了*”这种普通长句导致的目录解析灾难。
	
	* **自定义匹配：智能引导模式 (Smart Guided Mode)**：
	    针对不规则的异形章节名（如 `☆卷二 chap124 陨落☆`），提供无需编写代码的“向导式正则生成器”：
	    * **特征样本捕获**：在 UI 界面唤出专用弹窗，要求用户直接从源文件中复制并粘贴 1~2 个典型的异形章节行作为“训练样本”。
	    * **启发式正则自动推导 (Auto-Regex Deduction)**：
	        * **锚点定位算法**：后端接收样本后，自动扫描样本字符串，将“变化的部分”（如数字 `124` 或 `一百二十`）剥离出来，自动替换为正则的捕获组 `(\d+)` 或 `([一二三...零]+)`。
	        * **前缀/后缀转义**：将数字前后的固定字符（如 `chap`、`☆`）自动进行正则特殊符号转义，并拼装成完整的锚定行首的正则表达式（例如推导出：`^☆卷二\schap\s*(\d+).*?$`）。
	    * **沙箱即时校验与微调 (Real-time Validation Sandbox)**：
	        * **实时目录树抽取**：在弹窗右侧或下方提供一个预览面板。当正则生成后（或用户手动修改后），系统立即对预览切片（或后台全量文本）执行异步匹配，并将命中提取出的**纯标题以列表树的形式展示出来**。
	        * **可视化纠错**：用户可以通过查看抽取出的列表是否混入了奇怪的长句子，来直观判断推导出的正则是否精准。支持用户直接在输入框中修改系统推导出的正则表达式，下方列表列表将防抖（Debounce）实时更新。

*   **13.章节序号重排**：支持对识别出的章节进行重新顺序编号，修复序号跳变。
	* 章节输出结构
		Chapter
		{
		  index: int
		  title: str
		  line_number: int
		}

* **14.断句缝合**
	* **行尾状态特征识别 (Tail Feature Detection)**：
	    * **非终结符判定**：检测第 $N$ 行的最后一个非空字符。如果该字符是逗号（`，`）、顿号（`、`）、无标点汉字、或是英文字母，且**不属于**标准的句末终结符（如 `。` `！` `？` `”` `」` `…` `~`），则将该行标记为“高疑似断句行”。
	* **行首状态特征校验 (Head Feature Detection)**：
	    * **首字符校验**：检测第 $N+1$ 行的第一个非空字符。如果它以标点符号开头（尤其是 `，` `。` `”` 等原本应该连在上一行的符号），或者以非缩进的普通文字开头，则进一步确认缝合条件。
	    * **对话防误伤豁免**：若第 $N+1$ 行以左引号（`“` 或 `「`）开头，无论上一行状态如何，**绝对禁止**缝合，以保护角色对话的独立换行。
	* **缝合执行与安全边界 (Execution & Constraints)**：
	    * **长度阈值过滤**：为了防止将短标题、诗歌或列举项（如“第一卷”、“主角属性：”）错误合并，可设定一个**最短行宽阈值**（如：只有当第 $N$ 行字符数大于 15 个字时，才触发缝合检测）。
	    * **执行缝合**：满足上述所有安全条件后，移除第 $N$ 行与第 $N+1$ 行之间的换行符 `\n`。对于中文，直接首尾相连；若缝合处两侧均为英文字母，则自动补充一个半角空格 `\u0020` 以防单词粘连。
* 去广告 / 去网站信息（先不做）：
	*   内置常见小说网站广告正则库，并支持多维度自定义过滤（批量输入、正则/字面量切换、预览高亮）。（规则存储在本地 JSON）
### 2.3 输出文档与统计报告
* **15.输出文件**
	* 将内存中清洗完毕的最终 UTF-8 文本流，安全写入目标磁盘路径（包含生成无损备份文件的操作）

* **16. 生成统计报告**：
	* 收集流水线中各激活节点的执行日志（如：耗时、正则命中次数、剔除字符数），生成结构化处理报告，推送到前端 UI 并提供下载。


### 2.4 系统特性与性能

#### 2.4.2 操作安全与非破坏性处理 (Non-Destructive Pipeline)

本系统摒弃传统文本编辑器高耗能的 Undo/Redo 状态栈，采用“流水线批处理”与“原始文件只读”的隔离架构，确保数据绝对安全且内存占用极低。

* **源数据只读保护 (Read-Only Source)**：
    * 用户上传的原始 TXT 文件在系统内存/本地缓存中被标记为绝对只读。所有的预览切片和最终清洗生成，均从该只读源获取数据，杜绝源文件被意外污染。
* **声明式批处理执行 (Declarative Batch Processing)**：
    * 界面左侧的所有操作（转换编码、选定过滤字符、填写去广告正则等）仅修改配置状态树（`RuleConfig`），不直接修改全量文本。
    * 用户点击“开始完整清洗”后，系统按设定好的流水线规则一次性生成最终结果。
* **无状态重置与覆盖重做 (Stateless Regeneration)**：
    * **重做机制**：当用户对生成的最终 TXT 不满意时，左侧面板修改规则参数，点击重新生成。后端将自动释放上一次生成的临时文件结果，重新读取原始文件并应用新规则输出，实现极速覆盖。
    * **一键初始化**：提供“恢复默认配置”按钮，一键清空所有自定义正则与手动黑名单，将左侧配置树重置为初始状态，右侧预览区同步回归原始文本的初始展现形式。
### 2.4.3 异常捕获与系统监控 (Error Handling & Logging)

* **分级日志记录机制 (Hierarchical Logging)**：
    * **本地滚动存储**：后端 Python 使用标准 `logging` 模块，在可执行文件同级目录自动生成 `logs/` 文件夹。日志按天滚动存储（如 `app_2026-03-06.log`），避免单个日志文件过大。
    * **降噪与定级**：
        * `INFO` 级别：记录任务启停、耗时、核心配置项快照。
        * `WARNING` 级别：记录非致命异常，如“检测到大量不可解析乱码已执行跳过/替换”、“某条广告正则未能命中任何内容”。
        * `ERROR` 级别：记录系统崩溃、内存溢出或读写权限被拒的完整堆栈信息 (Traceback)。提供一键导出 `error.log` 的按钮。

* **场景化友好提示 (Contextual UI Prompts)**：
    系统需拒绝笼统的“处理失败”，而是根据错误码在 UI 顶端弹出不同类型的 Toast/Alert 提示，并自动重置 UI 状态以允许重试：
    * **物理文件异常**：检测到 0 字节文件、非文本二进制特征（如图片混入）、或被系统加密拒读时，提示：“无法读取：文件可能损坏、被加密或是非纯文本格式”。
    * **超限异常**：上传超过 50MB 的文件时，在前端直接拦截并提示：“文件过大：为保证处理性能，请将文本拆分至 50MB 以内”。
    * **空间不足异常**：保存输出文件时捕捉 `IOError`，提示：“导出失败：目标磁盘空间不足”。

### 2.5 交互反馈、洞察报告与安全存储 (Feedback, Reporting & Storage)
本模块负责清洗任务完成后的数据交付与状态闭环，设计原则为“轻量级渲染”与“无损保存”。
#### 2.5.1 多维度清洗洞察与报告下载 (Insight Report & Export)
* **结构化统计大屏**：任务结束后，UI 面板切换至结果页，直观展示本次处理的量化指标：
    * **基础指标**：总耗时、原始大小 vs 清洗后大小（如 `50MB -> 42MB`，直观展示“挤水分”效果）。
    * **深度指标**：章节识别总数、拦截非法控制字符频次、成功替换/剔除的广告与黑名单字符串次数。
* **处理报告独立导出**：
    * 提供“下载详细报告”按钮。前端将上述统计信息以及关键的正则命中记录（如：某条自定义去广告正则命中了多少次），打包生成一个轻量级的 `.json` 或 `.txt` 日志文件供用户下载。方便用户复盘清洗质量并优化自己的正则规则库。
```
{
  "file_size_before": 52428800,
  "file_size_after": 44000000,
  "processing_time": 12.4,
  "chapters_detected": 138,
  "removed_control_chars": 124,
  "removed_blacklist_chars": 45,
  "regex_hits": {
    "ad_rule_1": 32
  }
}
```

#### 2.5.2 差异对比与性能边界 (Diff View Guardrails)
* **局部切片对比约束**：
    * **性能红线**：为防止浏览器 DOM 树崩溃，**严禁在全量处理完成后对 50MB 级大文件执行全量 Diff 渲染**。
    * **复用机制**：界面上提供的 Split 或 Unified Diff 视图，仅在“配置调试阶段”对动态切片（如前 100 行或某指定章节的 100 行）生效。全量生成完毕后，视窗仅提供最终排版效果的纯文本预览（支持虚拟滚动渲染）。

#### 2.5.3 智能防御型保存机制 (Defensive Saving Mechanism)
由于工具支持作为本地服务运行处理本地文件，保存机制必须具备极高的容错性：
* **无损覆盖与静默备份 (Safe Overwrite)**：
    * 当用户选择“保存/覆盖原文件”时，后端**绝对不**直接执行覆写。而是先将原始文件重命名为 `[原文件名]_备份_[时间戳].txt`。
    * 确认备份成功落盘后，再将内存中的清洗结果以 UTF-8 格式写入原文件路径。
* **流式另存为 (Streamed Save As)**：
    * 支持自定义输出名称。对于 Web 交互，触发标准的浏览器大文件流式下载（Blob Download）；对于封装的 EXE 桌面级交互，唤起原生操作系统的目录选择器。
* **物理容量探针与异常熔断 (Capacity Probe & Fallback)**：
    * **写入前/写入中校验**：在后端执行磁盘写入操作时，若捕捉到操作系统抛出的 `IOError/ENOSPC` (No space left on device) 异常，系统立即触发熔断。
    * **脏数据清理与告警**：后端自动删除因空间不足而写入了一半的残缺文件（防止产生坏档）。同时通过 API 向前端传递致命错误，网页中心强制弹出高优级别的红色弹窗警报：“保存失败：目标磁盘剩余空间不足，操作已取消”。

### 2.6 规则配置持久化与资产管理 (Configuration & Asset Management)

为保障配置的安全性与交互的易用性，系统采用前后端分离的配置管理策略，杜绝要求用户手动操作底层文件。

* **配置存储与持久化 (Persistence Strategy)**：
    * **后端落盘**：用户的自定义规则（特别是繁杂的去广告正则库）统一由后端 FastAPI 序列化并存储在运行目录下的 `user_config.json` 文件中。
    * **前端状态挂载**：前端 Vue 在初始化时请求该配置，并挂载到全局状态库（如 Pinia）。所有的配置变更在点击“保存规则”后，通过 API 异步同步至后端覆写。

* **基于 UI 的无缝导入与导出 (UI-Driven Import/Export)**：
    * **导出（备份/分享）**：在左侧配置面板底部提供【导出配置】按钮。前端将其打包为标准的 `.json` 文件触发浏览器下载。方便深度用户备份自己的“极品清洗规则”，或分享给社区。
    * **导入（恢复/应用）**：提供【导入配置】上传区。用户上传 JSON 文件后，后端**必须引入 Schema 校验（如利用 Pydantic）**。一旦检测到缺少字段或格式错乱（如正则语法错误），立即拦截并提示，防止应用崩溃。

* **一键出厂重置与默认沙箱 (One-Click Factory Reset)**：
    * **不可变基准配置**：在 Python 源码内部或打包资源的只读目录中，封存一份绝对正确的 `default_config.json`（包含基础的章节匹配正则和通用广告黑名单）。
    * **安全重置机制**：当用户把正则写乱或配置损坏时，只需在 Web 界面点击【恢复默认配置】按钮。后端将读取这不可变的基准配置，直接覆盖当前的 `user_config.json` 并通知前端重绘，实现 1 秒“满血复活”。

## 5. UI界面设计
### UI 架构与交互设计说明 (2/3 + 1/3 混合式布局)

#### 1. 架构设计理念
系统前端采用非对称的 **2:1 (66vw : 33vw)** 黄金分割响应式布局。
- **左侧操作流 (2/3)**：承载核心步进式业务逻辑（上传 -> 配置 -> 深度比对 -> 下载）。
- **右侧反馈流 (1/3)**：实时预览视窗，展示规则应用后的效果并辅以 Diff 高亮。

#### 2. 大文件加载交互
针对大文件处理，引入基于 SSE 的四步交互流：
- **任务注册**：前端 POST 上传文件，后端返回 `task_id`。
- **SSE 通道建立**：前端监听 `/api/stream-progress/{task_id}`。
- **后端流式 Yield**：FastAPI 分块处理并实时 Yield 进度百分比。
- **UI 更新**：前端接收数据实时驱动进度条动画。

#### 3. 右侧预览区：动态切片与章节跳转
- **预览工具栏**：固定悬浮的下拉选择器。
- **按需切片加载**：支持用户选择章节跳转，前端携带行号锚点向后端发起局部请求，实现高性能渲染。Diff 视图仅用于**预览样本（片段）**，而非全量文件

## 8.文本处理pipeline
本系统核心引擎采用动态流水线（Dynamic Pipeline）架构。文本数据的整个处理生命周期被严格划分为三个阶段。其中，阶段二的执行链路完全由前端传递的 `RuleConfig` JSON 状态树驱动。系统会按照以下既定顺序执行，**当遇到用户未勾选（即状态为 `false`）的处理节点时，数据流将直接穿透跳过（Skip）该节点，进入下一环节。**

### 阶段一：编码与基础清理（强制执行）
此阶段为底层数据的安全底座，负责将各种杂乱的物理文件统一转化为标准化的内存字符串。不受用户 UI 勾选状态影响，必须执行。
- **1. 读取文件**
- **2. 编码检测**
- **3. UTF-8 转换**
- **4. BOM 去除**

### 阶段二：动态清洗组装线（条件执行）
此阶段为核心业务逻辑区。系统读取用户的勾选配置，按顺序执行。未勾选的项目直接跳过，零性能损耗。
- **[ ] 5. 换行符归一化**
- **[ ] 6. HTML 清理**
- **[ ] 7. 非法字符过滤**
- **[ ] 8. 手动过滤非法字符**
- **[ ] 9. 全半角转换**
- **[ ] 10. 段落清洗**
- **[ ] 11. 空行压缩**
- **[ ] 12. 章节识别**
- **[ ] 13. 序号重排**
- **[ ] 14. 断句缝合**

### 阶段三：输出文档与统计报告（强制执行）
经过一系列清洗与跳过后，进入数据落盘与闭环收尾。
- **15. 输出文件**
- **16. 生成统计报告**
